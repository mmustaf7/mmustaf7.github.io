How DNS Works

History of DNS

Before DNS, Internet-connected systems used a centralized file called HOSTS.TXT to resolve domain names. This approach became impractical as the network grew. In 1983, Paul Mockapetris designed the Domain Name System as a scalable, distributed solution. DNS introduced a hierarchical model for name resolution and was standardized in RFC 882 and RFC 883. It laid the foundation for the modern, scalable naming architecture of the Internet.

DNS Structure and Hierarchy

DNS operates as a tree-like structure:

  • Root Domain: Represented by a dot (.) at the top
  • Top-Level Domains (TLDs): Examples: .com, .org, .edu, .uk
  • Second-Level Domains: Example: example in example.com
  • Subdomains: Example: mail.example.com

The DNS system is overseen by ICANN and IANA, which coordinate root zones and delegate authority to domain registrars.

DNS Records

Each DNS zone contains different record types:

  • A: Maps a domain to an IPv4 address
  • AAAA: Maps to an IPv6 address
  • CNAME: Canonical name (alias to another domain)
  • MX: Mail exchange server record
  • NS: Nameserver record indicating authoritative DNS servers
  • TXT: Text data used for SPF, DKIM, domain verification

How DNS Resolution Works

When a user types www.example.com into a browser, the following steps occur:

  1. The browser checks its DNS cache.
  2. If not found, the request is sent to a recursive DNS resolver (often provided by the ISP or a public DNS service).
  3. The resolver queries a Root DNS server, which directs it to the correct TLD server.
  4. The TLD server then provides the address of the authoritative name server for the domain.
  5. The authoritative server replies with the final IP address.
  6. The resolver caches the result and returns it to the browser.

This multi-step process typically takes milliseconds and enables users to access websites by name instead of IP address.

DNS Caching

To improve performance, DNS uses caching at multiple levels:

  • Browser Cache
  • Operating System Cache
  • Recursive Resolver Cache

Each DNS record has a Time-To-Live (TTL) that determines how long it remains cached. Lower TTLs make updates propagate faster but increase DNS traffic.
